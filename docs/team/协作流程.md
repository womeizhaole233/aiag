# 团队协作流程

本文面向非程序团队，目标是让策划、编剧、美术、学术校核和小游戏负责人都能用统一方式交付内容。

## 一、总负责人规则

总负责人保留 `main` 的最终合并权。

`main` 的含义：

1. 当前可以运行。
2. 当前可以给团队查看。
3. 当前口径已经确认。
4. 当前内容属于 M1 主线范围。

## 二、成员如何工作

成员不直接修改 `main`。每项任务单独开分支。

当前仓库按四类分支理解：

| 分支类型 | 用途 |
|---|---|
| `main` | 稳定、可运行、口径已确认 |
| `develop` | 多模块整合测试，准备进入 `main` |
| `feature/*` | NPC、小游戏、记录夹、自由对话 API 等模块开发 |
| `task/*` | 文档修订、素材清单、路径修正、归档整理等小任务 |

详细分类见：`docs/team/Git分支与文档分类.md`。

推荐分支格式：

```text
类型/任务名称
```

示例：

```text
docs/prd-cleanup
content/clue-table-v1-1
asset/rear-room-images
feature/npc-fixed-dialogue
feature/minigame-dig
archive/old-docs-cleanup
```

## 三、不同模块的交付边界

| 模块 | 可以改什么 | 不应该改什么 |
|---|---|---|
| 图片 / 素材 | `assets/`、图片资产表 | 不直接改线索结论 |
| 线索 | 线索总表、ID 字典、章节线索稿 | 不擅自加入未校核剧情推断 |
| 固定剧情 NPC | 剧情节点表、NPC 对话、选项反馈 | 不把虚构内容写成考古事实 |
| 自由对话 NPC | NPC 知识范围、暂不透露边界、回复规则 | 不改主线推进条件 |
| 小游戏 | 独立原型、接入表、产出线索说明 | 不脱离 M1 主线做小游戏合集 |
| 程序 / 整合 | `index.html`、`styles.css`、`game.js`、`data/scenes.js` | 不修改未经确认的学术结论 |

## 四、每次交付必须写清楚

每次提交或交付应说明：

1. 这次改了什么。
2. 对应哪个章节。
3. 引用了哪些 `clueId`、`imageId` 或 PRD 条目。
4. 是否影响主线推进。
5. 是否需要学术校核或总负责人确认。

## 五、合并前检查

合并到 `main` 前至少检查：

1. 本地是否可以运行。
2. 图片路径是否存在。
3. 文档是否和线索总表、ID 字典冲突。
4. 是否混入旧口径。
5. 是否有日志、截图、zip、IDE 配置等临时文件。

## 六、给不熟悉 Git 的成员

如果成员不会使用 Git，可以只交付固定格式文档或表格。由总负责人、程序整合人或 AI 助手把内容整理进对应分支。

不要让成员把所有内容直接丢进根目录；每个模块都应该有明确入口和负责人。
