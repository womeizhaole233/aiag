# 团队分工与模块接口需求文档

> 本文用于约束当前半个月开发周期内的团队分工、交付格式、模块边界与接入方式。目标是让图片、线索、剧情、NPC、小游戏分别推进时，最终能够被主项目和 AI 清楚理解并顺利整合。

## 1. 当前团队分工判断

当前分工：

| 人员方向 | 人数 | 当前职责 |
|---|---:|---|
| 图片建模 + 线索 | 1 人 | 整理图片、处理视觉资产、建立线索对应关系 |
| 固定剧情 NPC | 1 人 | 编写确定发生的剧情节点、NPC 对话、玩家选择与反馈 |
| 自由对话 NPC | 1 人 | 设计 API 对话所需的 NPC 知识范围、暂不透露边界和回复规则 |
| 独立小游戏开发 | 2 人 | 开发少量核心小游戏原型，并说明接入主线的方式 |

这个分工总体可行，但需要调整一个关键点：

**NPC 剧情和线索不能完全分开。**

原因是：NPC 说什么、什么时候出现、玩家选项如何反馈，都必须依赖线索。否则剧情会变成文学文本，线索会变成资料表，两者无法自然接入游戏。

因此建议采用：

```text
图片建模 + 线索负责人：负责“事实线索”和素材依据。
固定剧情 NPC 负责人：负责“确定发生的章节剧情、选项和反馈”。
自由对话 NPC 负责人：负责“玩家主动提问时，API 应如何扮演 NPC”。
小游戏负责人：负责“操作玩法”和产出结果。
主策划/整合者：负责判断哪些内容进入主线，统一字段和接入规则。
```

其中，线索表必须有明确负责人。建议由“图片建模 + 线索”成员作为线索表主负责人，主策划/整合者负责审核哪些线索是当前单线流程必需内容；固定剧情、自由对话和小游戏负责人都引用线索表中的 `clueId`，不直接自行新增流程必需线索。

## 2. 半个月开发目标

半个月内不建议追求完整大版本，应完成一个“可整合的内容生产体系”。

### 必须完成

1. M1 主线 PRD 固定。
2. 每个模块都有统一交付表。
3. NPC 剧情节点能对应具体场景和线索。
4. 第一版只保留三个核心小游戏，并能说明触发条件、成功结果和产出线索。
5. 图片和线索能追溯到原始材料。
6. 所有人交付内容都能被 AI 直接读懂并接入。

### 暂不追求

1. 完整 M3 开发。
2. 完整剧情分支。
3. 所有小游戏接入主项目。
4. 所有图片穷尽式使用。
5. 复杂多结局。

## 3. 总模块划分

| 模块 | 负责人方向 | 核心职责 | 最终交付 |
|---|---|---|---|
| 图片资产模块 | 图片建模 + 线索 | 整理底图、近景图、证据图、备用图 | 图片资产清单 |
| 线索模块 | 图片建模 + 线索 | 建立线索编号、来源、层级、对应图片 | 线索总表 |
| NPC 固定剧情模块 | 固定剧情 NPC | 写章节剧情、NPC 对话、选项反馈 | NPC 剧情节点表 |
| NPC 自由对话模块 | 自由对话 NPC / 主项目整合 | 设计可被 API 调用的受控 NPC 问答 | NPC 自由对话设定表 |
| 小游戏模块 | 小游戏开发 | 做可独立测试的小游戏，并说明接入规则 | 小游戏接入表 |
| 场景接入模块 | 主项目整合 | 把图片、线索、剧情、小游戏接入主流程 | 场景数据表 |
| 记录夹/结论模块 | 主项目整合 | 统一线索记录、复查、汇总和章节完成 | 章节完成条件表 |

## 4. 模块之间的“接口”应该怎么写

这里的“接口”不一定是程序代码。对当前团队来说，接口主要是**固定格式的交付表**。

每个成员只要按表交付，AI 和主项目就能知道：

```text
这个内容属于哪个章节？
它在哪张图上出现？
由什么触发？
玩家做什么？
完成后产生什么线索？
是否影响进入下一章？
```

## 5. 图片资产模块接口

图片负责人需要交付一张图片资产表。

### 表格字段

| 字段 | 说明 |
|---|---|
| imageId | 图片唯一编号，例如 `M1_front_west_main` |
| 原始路径 | 原图在资料夹中的位置 |
| 项目路径 | 复制到项目后的路径，例如 `assets/M1/...` |
| 图片名称 | 原始文件名 |
| 所属章节 | 墓门 / 甬道 / 前室 / 过道 / 后室 |
| 所属视角 | 西壁 / 东壁 / 顶部 / 入口总览等 |
| 图片类型 | 主底图 / 近景图 / 证据图 / 备用图 |
| 对应线索 | 可关联的 clueId |
| 是否已使用 | 是 / 否 |
| 备注 | 重复图、质量问题、待裁剪、待补图等 |

### 示例

| imageId | 所属章节 | 所属视角 | 图片类型 | 对应线索 |
|---|---|---|---|---|
| M1_gate_main | 墓门 | 正面 | 主底图 | gate_lintel, gate_bricks |
| M1_front_east_music | 前室 | 东壁 | 近景图 | front_music_group |

## 6. 线索模块接口

线索负责人需要交付一张线索总表。线索必须以真实材料为依据，并明确每条线索在当前单线流程中的作用。

线索表不是可有可无的资料整理，而是剧情、小游戏和场景接入的共同依据。没有进入线索表的内容，不能直接作为流程推进条件。

### 表格字段

| 字段 | 说明 |
|---|---|
| clueId | 游戏内线索编号 |
| sourceFile | 来源文件 |
| sourceClueId | 材料中的线索编号 |
| 所属章节 | 墓门 / 甬道 / 前室 / 过道 / 后室 |
| 所属图片 | imageId |
| 线索层级 | P0 / P1 / P2 / H / E / R / F |
| 事实描述 | 可观察事实，2-3 句 |
| 记录文案 | 进入记录夹的短句 |
| 是否影响推进 | 是 / 否 |
| 关联线索 | 可关联的其他 clueId |
| 注意事项 | 是否易误判、是否需要降级、是否只作支线 |

### 线索分层要求

| 层级 | 含义 |
|---|---|
| P0 | 核心推进线索 |
| P1 | 关键解释线索 |
| P2 | 沉浸观察线索 |
| H | 误导或待验证线索 |
| E | 环境线索 |
| R | 推理关系或章节结论 |
| F | 剧情虚构线索 |

### 单线流程线索划分

| 类型 | 当前流程作用 | 使用方式 |
|---|---|---|
| P0 | 必需 | 必须获得，影响章节推进 |
| P1 | 重要 | 支撑章节结论，可进入记录夹复查 |
| P2 | 补充 | 增加探索感，不影响推进 |
| H | 待验证 | 误导或需要降级 |
| E | 环境 | 用于定位和空间理解 |
| R | 章节完成后生成 | 作为结论卡或关系链 |
| F | 剧情 | 只进入剧情线，不作为考古证据 |

判断一条线索是否为当前流程必需线索，可以使用下面的问题：

```text
如果删除这条线索，玩家是否仍能理解本章结论或进入下一场景？
如果不能，它可能是 P0 或 P1。
如果能，它更适合做 P2、H、E 或 F。
```

### 线索负责人权限

| 事项 | 负责人 |
|---|---|
| 新增真实线索 | 线索负责人 |
| 判断是否为当前流程必需 | 主策划/整合者 + 线索负责人 |
| 引用线索写 NPC 对话 | 剧情/NPC 负责人 |
| 引用线索设计小游戏产出 | 小游戏负责人 |
| 新增剧情虚构线索 | 剧情/NPC 负责人提出，主策划审核 |
| 修改 P0 线索 | 必须经过主策划确认 |

## 7. NPC 固定剧情模块接口

固定剧情 NPC 负责人不能只交文学稿，需要交付“剧情节点表”。固定剧情指游戏中确定会发生的章节进入、任务说明、玩家选择、小游戏前后引导和章节收束。

### 表格字段

| 字段 | 说明 |
|---|---|
| nodeId | 剧情节点编号 |
| 所属章节 | 墓门 / 甬道 / 前室 / 过道 / 后室 |
| 触发条件 | 首次进入 / 点击线索后 / 缺项时 / 完成章节后 |
| 出场 NPC | 村民 / 技工 / 队员 / 教授等 |
| 节点目的 | 说明目标 / 提醒观察 / 反馈选择 / 收束结论 |
| 对话正文 | NPC 和玩家对话 |
| 玩家选项 | 选项 A/B/C |
| 正确反馈 | 玩家选择合理时的反馈 |
| 错误反馈 | 玩家选择不合理时的提示 |
| 产出结果 | 新增记录 / 解锁小游戏 / 解锁场景 / 仅剧情 |
| 绑定线索 | clueId |
| 是否影响推进 | 是 / 否 |

### NPC 内容原则

1. NPC 不直接告诉玩家本章 R 结论。
2. NPC 负责提醒观察方向、解释工作方法、引导复查。
3. NPC 的话必须能对应场景或线索。
4. 虚构剧情和真实考古事实要分开标注。
5. 第一版不做复杂分支，选择只影响反馈，不改变主线结论。

## 8. 小游戏模块接口

小游戏负责人需要交付“小游戏接入表”，不能只交 HTML 文件。

当前半个月周期内，小游戏数量必须收束。第一版只保留三个核心小游戏：

| miniGameId | 小游戏 | 所属阶段 | 状态 |
|---|---|---|---|
| mg_dig_match3 | 挖土消消乐 | 墓外 / 墓门前 | 保留 |
| mg_inscription_reading | 题记辨读 | 过道 | 保留 |
| mg_rear_relic_position | 后室遗物定位 | 后室 | 保留 |

墓门文字解谜、甬道纹样校准、文物擦灰、器物归类、终章线索墙暂不进入第一版开发，只保留为后续方案。

### 表格字段

| 字段 | 说明 |
|---|---|
| miniGameId | 小游戏编号 |
| 小游戏名称 | 挖土消消乐 / 题记辨读 / 后室遗物定位 |
| 所属章节 | 墓外 / 过道 / 后室 |
| 触发条件 | 玩家获得哪些线索后触发 |
| 入口方式 | 点击哪个热区或按钮 |
| 玩家操作 | 玩家具体做什么 |
| 成功条件 | 如何判定完成 |
| 失败处理 | 是否允许重试，NPC 是否提示 |
| 产出线索 | 完成后新增的 clueId |
| 解锁结果 | 解锁场景 / 解锁近景 / 解锁结论 / 仅奖励 |
| 使用素材 | 图片、音效、字体等 |
| 独立运行方式 | 如何本地打开测试 |
| 接入备注 | 需要主项目提供什么数据 |

### 小游戏接入原则

1. 小游戏必须对应考古动作。
2. 小游戏完成后必须有明确产出。
3. 小游戏不能只为了好玩而存在。
4. 第一版小游戏应允许跳过或重试，避免卡死主线。
5. 小游戏文案也要遵守线索文案规范。
6. 小游戏第一版总数不超过三个。

### 小游戏交付验收条件

| 小游戏 | 必须交付 | 完成结果 |
|---|---|---|
| 挖土消消乐 | 可独立运行原型、方块素材、接入表、胜利条件说明 | 完成后产出清理记录，开放墓门区域 |
| 题记辨读 | 题记图片或拓片素材、可选字/辨读逻辑、接入表 | 完成后产出过道题记辨读记录 |
| 后室遗物定位 | 后室分布图或平面图、可定位对象、正确位置规则、接入表 | 完成后产出后室遗存关系记录 |

每个小游戏必须说明触发条件、入口位置、玩家操作、成功条件、失败处理、完成后是否写入记录夹、是否解锁下一节点、使用哪些 `clueId`。

## 8.1 NPC 自由对话模块接口

如果要加入“点击 NPC 后通过 API 自由对话”，自由对话 NPC 负责人需要交付四张表：

| 表 | 作用 |
|---|---|
| NPC 自由对话设定表 | 定义哪些 NPC 可自由对话、在哪些章节出现、说话风格是什么 |
| NPC 知识范围表 | 定义每个 NPC 在每章可以引用哪些 `clueId` 和概念 |
| NPC 暂不透露 / 边界表 | 定义不能提前剧透或无依据生成的内容 |
| NPC 默认回复规则 | 定义玩家问错、问空、问剧透、问无关内容时如何回答 |

第一版建议只开放一个自由对话 NPC：考古领队 / 教授。

运行逻辑：

```text
玩家点击“询问领队”
→ 输入问题
→ 前端提交 npcId、sceneId、viewId、collectedClues、missingClues、playerQuestion
→ 后端调用 API
→ 返回 NPC 回答
→ 前端显示
→ 默认不写入记录夹，不直接推进流程
```

自由对话 NPC 不得：

1. 自行创造新的考古事实。
2. 提前透露后续章节结论。
3. 直接告诉玩家所有答案。
4. 默认写入记录夹。
5. 由前端直接暴露 API Key。

详细规则见《NPC自由对话模块需求文档.md》。

## 9. 场景接入模块接口

主项目最终需要把各模块接入场景数据。

每个场景至少需要以下信息：

| 字段 | 说明 |
|---|---|
| sceneId | 场景编号，例如 `front_chamber` |
| viewId | 视角编号，例如 `front_west` |
| imageId | 当前视角使用的图片 |
| hotspotId | 热区编号 |
| hotspotType | clue / navigation / closeup / miniGame / npc |
| target | 点击后进入哪个 view、小游戏或剧情节点 |
| clueId | 如果是线索热区，对应哪个线索 |
| requiredClues | 如果是出口，需要哪些线索 |
| requiredActions | 是否需要记录夹汇总或小游戏完成 |
| lockedMessage | 条件不足时提示什么 |

### 第 7-10 天：主项目接入一小段闭环

建议只做一个可验证闭环：

```text
墓外或墓门前
→ NPC 开场
→ 挖土消消乐
→ 获得线索
→ 进入墓门
```

目标不是做全，而是验证：

1. 剧情节点能触发。
2. 小游戏能被打开。
3. 小游戏结果能写入记录或推进条件。
4. AI 能读懂各模块数据。

### 第 11-14 天：扩展到 M1 已有主线

| 任务 | 目标 |

| 接入自由对话 NPC | 验证玩家能主动询问领队并获得受控提示 |
| 检查文案风格 | 确保剧情、线索、小游戏不互相打架 |

## 11. 每个人的交付要求

### 图片建模 + 线索负责人

必须交付：

1. 图片资产表。
2. 线索总表。
3. 主线线索与非主线线索划分。
4. 每条线索对应图片。
5. 每张图片建议用途。
6. 哪些图片暂不使用。
7. 哪些剧情线索不能作为考古证据。

不能只交：

```text
一堆图片文件
一堆没有编号的说明文字
```

### 固定剧情 NPC 负责人

必须交付：

1. NPC 角色表。
2. 固定剧情节点表。
3. 玩家选项与反馈。
4. 每个剧情节点绑定的章节和线索。
5. 小游戏前后引导文本。
6. 章节进入和章节收束文本。
7. 哪些内容是虚构剧情，哪些内容是考古事实。

不能只交：

```text
长篇小说式剧情文本
没有触发条件的对白
没有对应线索的角色发言
```

### 自由对话 NPC 负责人

必须交付：

1. NPC 自由对话设定表。
2. NPC 知识范围表。
3. NPC 暂不透露 / 边界表。
4. NPC 默认回复规则。
5. 示例问答至少 10 条。
6. 每个章节的 currentGoal。
7. 每个章节可引用的 clueId。
8. 每个章节 forbiddenTopics。

不能只交：

```text
泛泛的人设说明
没有知识范围的 API 提示词
没有暂不透露边界的自由聊天设定
让 NPC 可以自行编造考古事实
```

### 小游戏负责人

必须交付：

1. 可运行小游戏原型。
2. 小游戏接入表。
3. 触发条件。
4. 成功条件。
5. 完成后产出的线索或推进结果。
6. 引用的 `clueId`，不能自行创造主线线索。

不能只交：

```text
独立 HTML 小游戏
没有说明如何接入主线的玩法
```

## 13. AI 接入时最需要的信息

每次让 AI 接入一个模块时，必须提供：

```text
1. 这个模块属于哪个章节。
2. 它使用哪些图片。
3. 它绑定哪些线索。
4. 玩家如何触发。
5. 完成后发生什么。
6. 是否写入记录夹。
7. 是否影响进入下一章。
8. 是否有真实材料依据。
```

如果缺少这些信息，AI 很容易出现：

1. 胡乱生成文案。
2. 把虚构剧情当作真实线索。
3. 把小游戏做成独立功能。
4. 导航、记录、完成条件互相冲突。
5. 接入后玩家不知道该做什么。

## 14. GitHub 协作规则

当前团队不建议所有人直接把文件提交到 `main`。

### 14.1 main 与 branch 的含义

| 名称 | 含义 | 使用方式 |
|---|---|---|
| main | 主分支、稳定版本 | 只放已经确认可整合的内容 |
| branch | 功能分支、个人工作线 | 每个人或每个模块在自己的分支上工作 |

`main` 不等于“主线剧情”，它是 GitHub 里的稳定代码分支。当前游戏只有一条流程线，但 Git 协作仍然需要分支。

### 14.2 推荐分支

| 成员方向 | 推荐分支名 |
|---|---|
| 图片 + 线索 | `clue-assets` |
| 剧情 / NPC A | `npc-story-a` |
| 剧情 / NPC B | `npc-story-b` |
| 小游戏 A | `minigame-dig` |
| 小游戏 B | `minigame-gate` |
| 主项目整合 | `integration-mvp` |

### 14.3 提交原则

1. 所有人都可以把自己的工作提交到 GitHub，但不要直接推到 `main`。
2. 每个人应提交到自己的 branch，或者提交到指定的模块文件夹。
3. 进入 `main` 前，需要由主策划/整合者检查字段、路径、线索编号和运行效果。
4. 独立小游戏可以先放在自己的文件夹或分支中，但必须附带小游戏接入表。
5. 剧情文档可以先放在文档目录中，但必须附带剧情节点表。
6. 图片和线索可以先放在素材目录和线索表中，但必须有 imageId、clueId 和来源说明。

### 14.4 不建议的做法

```text
所有人都直接改 main。
只上传一堆图片，没有图片资产表。
只上传一篇剧情，没有节点表。
只上传小游戏 HTML，没有接入说明。
直接把 M3 内容混进 M1 当前流程。
```

## 15. 最重要的协作原则

1. **先表格，后代码。**
2. **先 M1，后 M3。**
3. **先闭环，后扩展。**
4. **先真实线索，后剧情包装。**
5. **小游戏必须有考古动作依据。**
6. **NPC 必须服务章节目标。**
7. **所有交付都必须说明触发条件和完成结果。**

## 16. 结论

当前分工可以保留，但需要增加“统一字段表”和“模块接入表”。

在半个月周期内，最合理的目标不是把所有东西都做完，而是完成一个稳定协作框架：

```text
图片能找到出处
线索能对应图片
剧情能绑定线索
小游戏能产出结果
主项目能识别这些内容
AI 能根据表格稳定接入
```

只要这个框架建立起来，后续继续扩展 M1、M2、M3 都不会乱。
