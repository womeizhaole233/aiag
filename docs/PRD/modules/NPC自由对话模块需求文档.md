# NPC 自由对话模块需求文档

> 本文用于定义“点击 NPC 后通过 API 进行自由对话”的交付规则、运行逻辑和安全边界。该模块服务于玩家提示、陪伴、复查和剧情沉浸，不直接替代固定剧情节点和考古线索系统。

## 1. 模块定位

NPC 交互分为两类：

| 类型 | 作用 | 是否控制流程 |
|---|---|---|
| 固定剧情 NPC | 章节进入、关键节点、选项反馈、章节收束 | 可以控制流程 |
| 自由对话 NPC | 玩家主动提问、轻提示、术语解释、线索复查 | 默认不控制流程 |

自由对话 NPC 的核心原则：

```text
固定剧情负责推进，自由对话负责提示和陪伴。
```

自由对话 NPC 不能直接告诉玩家本章 R 结论，不能凭空生成考古事实，不能提前剧透后续章节。

## 2. 第一版范围

半个月 MVP 中不建议所有 NPC 都支持自由对话。第一版只建议开放 1 个 NPC：

| npcId | NPC | 位置 | 功能 |
|---|---|---|---|
| npc_professor | 考古领队 / 教授 | 记录夹或章节入口总览 | 回答当前目标、缺少什么、如何复查线索 |

其他 NPC 可先只做固定剧情节点，后续再扩展自由对话。

## 3. NPC 自由对话交付内容

剧情 / NPC 负责人需要提交以下四类内容：

1. NPC 自由对话设定表。
2. NPC 知识范围表。
3. NPC 暂不回答 / 边界表。
4. NPC 默认回复规则。

### 3.1 NPC 自由对话设定表

| 字段 | 说明 |
|---|---|
| npcId | NPC 编号 |
| NPC 名称 | 例如粟柏年、周淼、赵老倔 |
| 身份 | 教授 / 队员 / 技工 / 村民 |
| 可对话章节 | 墓外 / 墓门 / 甬道 / 前室 / 过道 / 后室 |
| 出现位置 | 记录夹、入口总览、章节收束后等 |
| 对话功能 | 提醒目标 / 解释术语 / 引导复查 / 氛围陪伴 |
| 说话风格 | 学术、克制、口语、经验型、同伴式 |
| 是否影响推进 | 第一版建议填“否” |
| 是否写入记录夹 | 默认否 |

### 3.2 NPC 知识范围表

自由对话 NPC 只能知道被允许知道的内容。

| 字段 | 说明 |
|---|---|
| npcId | NPC 编号 |
| 章节 | 当前章节 |
| 允许引用的 clueId | 可引用哪些线索 |
| 允许引用的 nodeId | 可引用哪些剧情节点 |
| 允许解释的概念 | 例如封门砖、叠涩、地券、妇人启门 |
| 可提示的方向 | 可以提醒玩家观察哪里 |
| 不能跨越的章节 | 不能提前说出哪些后续内容 |

示例：

| npcId | 章节 | 允许引用的 clueId | 可提示方向 | 不能跨越 |
|---|---|---|---|---|
| npc_professor | 墓门 | gate_L01, gate_L02, gate_L03 | 门额、封门砖、门前地面 | 不提前说明后室地券 |
| npc_professor | 过道 | passage_L01, passage_L02, passage_L03 | 纪年题记、破子棂窗、顶部宝盖 | 不提前总结后室结论 |

### 3.3 NPC 暂不回答 / 边界表

| 字段 | 说明 |
|---|---|
| npcId | NPC 编号 |
| 阶段 | 当前章节或剧情阶段 |
| 暂不回答内容 | 不能提前说出的结论或尚未分栏的推测 |
| 边界原因 | 剧透 / 无材料依据 / 需要玩家先观察 |
| 推荐替代说法 | 可以如何委婉提示 |

示例：

| npcId | 阶段 | 暂不回答内容 | 推荐替代说法 |
|---|---|---|---|
| npc_professor | 墓门 | 直接告诉玩家点击哪几个热区 | “先把门额和封门砖放在一起看。” |
| npc_professor | 前室 | 直接给出礼仪空间结论 | “你可以先把人物、器物和顶部结构分开记录。” |

### 3.4 NPC 默认回复规则

| 玩家问题类型 | 回复策略 |
|---|---|
| 问当前目标 | 结合当前章节目标回答 |
| 问还缺什么 | 根据已收集 clueId 和章节条件给轻提示 |
| 问某个术语 | 用简短考古解释回答 |
| 问后续剧透 | 拒绝剧透，引导回当前场景 |
| 问无关问题 | 用角色口吻拉回现场 |
| 问材料外结论 | 提醒需要更多证据 |
| 问虚构剧情 | 标明这是传闻或个人线索，不作为考古证据 |

## 4. 运行逻辑

### 4.1 玩家交互流程

```text
玩家点击 NPC 或“询问领队”
→ 打开自由对话输入框
→ 玩家输入问题
→ 前端收集当前上下文
→ 后端调用 API
→ API 生成 NPC 口吻回答
→ 前端显示回答
→ 默认不写入记录夹
```

### 4.2 API 输入上下文

每次调用 API，应提供以下信息：

| 字段 | 说明 |
|---|---|
| npcId | 当前 NPC |
| npcName | NPC 名称 |
| npcRole | NPC 身份 |
| npcTone | 说话风格 |
| sceneId | 当前章节 |
| viewId | 当前视角 |
| currentGoal | 当前章节目标 |
| collectedClues | 玩家已收集 clueId |
| requiredClues | 当前章节必需 clueId |
| missingClues | 当前还缺哪些 clueId |
| allowedClues | NPC 当前可引用的 clueId |
| forbiddenTopics | 暂不回答内容，字段名可沿用但含义是“本章回答边界” |
| recentDialog | 最近几轮对话 |
| playerQuestion | 玩家输入 |

### 4.3 API 输出格式

API 返回结果建议采用固定结构，方便前端处理：

```json
{
  "npcId": "npc_professor",
  "reply": "先别急着往里走。把门额和封门砖的位置关系再看一遍，记录夹里应该能找到线索。",
  "hintLevel": "light",
  "referencedClues": ["gate_L01", "gate_L02"],
  "shouldSaveToJournal": false,
  "suggestedAction": "open_journal"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| reply | 显示给玩家的 NPC 回答 |
| hintLevel | none / light / direct |
| referencedClues | 回答引用了哪些 clueId |
| shouldSaveToJournal | 是否写入记录夹，默认 false |
| suggestedAction | 可选，建议玩家打开记录夹、回到某视角等 |

第一版建议只使用 `reply`，其他字段作为预留。

## 5. 写入记录夹规则

自由对话默认不写入记录夹。

只有以下情况可以写入：

1. NPC 引导玩家发现一个已经存在的 `clueId`。
2. 玩家完成一个明确剧情节点。
3. 主策划确认该 NPC 提示需要保留为“NPC 提示记录”。

即使写入，也必须标为：

```text
NPC 提示
```

不能标为：

```text
考古证据
```

## 6. 与固定剧情节点的区别

| 项目 | 固定剧情 NPC | 自由对话 NPC |
|---|---|---|
| 触发方式 | 到达节点自动触发或点击固定热区 | 玩家主动点击 NPC 并输入问题 |
| 内容来源 | 预先写好的剧情节点表 | API 根据上下文生成 |
| 是否可控 | 高 | 需要规则约束 |
| 是否推进流程 | 可以 | 第一版默认不推进 |
| 是否写入记录夹 | 视节点而定 | 默认不写入 |
| 风险 | 低 | 可能剧透、编造、偏题 |

## 7. 前端交互建议

第一版 UI 不应做复杂聊天系统，可以采用轻量弹窗：

```text
点击“询问领队”
→ 出现输入框
→ 玩家输入一句话
→ 显示 NPC 回答
→ 可继续问 / 关闭
```

不建议第一版加入：

1. 多 NPC 同屏聊天。
2. 长期聊天记录面板。
3. 复杂好感度。
4. NPC 自动频繁打断玩家。
5. 自由对话直接解锁关键场景。

## 8. 后端与安全要求

如果接入 API，不应在前端暴露 API Key。

推荐结构：

```text
前端游戏
→ 后端 / serverless 接口
→ 调用大模型 API
→ 返回 NPC 回答
→ 前端显示
```

禁止：

```text
HTML / JS 前端直接写 API Key 调用模型
```

## 9. 与线索表的关系

自由对话 NPC 必须引用线索表。

剧情组可以设计 NPC 的说话风格和回复边界，但不能让 NPC 自行创造新的考古事实。

小游戏组如果需要 NPC 在小游戏前后给提示，也必须引用对应 `clueId` 或 `miniGameId`。

## 10. MVP 接入建议

第一版只做一个最小闭环：

```text
玩家在墓门或记录夹中点击“询问领队”
→ 输入“我现在还缺什么？”
→ 系统根据 collectedClues 和 missingClues 生成提示
→ NPC 提醒玩家回看某个区域
→ 不直接解锁场景
```

验收标准：

1. 玩家能主动打开自由对话。
2. NPC 回答符合角色口吻。
3. NPC 能基于当前章节和已收集线索回答。
4. NPC 不提前剧透。
5. NPC 不生成材料外事实。
6. 默认不写入记录夹。
7. API Key 不暴露在前端。

## 11. 交付清单

剧情 / NPC 负责人需要提交：

1. NPC 自由对话设定表。
2. NPC 知识范围表。
3. NPC 暂不回答 / 边界表。
4. NPC 默认回复规则。
5. 示例问答 10 条。

技术 / 整合负责人需要提交：

1. 自由对话入口位置。
2. API 调用上下文格式。
3. API 返回格式。
4. 前端显示方式。
5. 是否写入记录夹的规则。
6. 错误处理方式，例如网络失败时显示固定回复。

线索负责人需要提交：

1. 可被 NPC 引用的 clueId 列表。
2. 每个章节的 currentGoal。
3. 每个章节的 requiredClues。
4. 每个章节的 forbiddenTopics。
