# 仓库整理方案

本文用于指导本地仓库分类、文档归位和后续 Git 协作。当前目标不是一次性重构全部项目，而是先把仓库变成“可读、可运行、可协作”的状态。

## 一、整理原则

`main` 应该代表当前稳定版本，而不是所有人的草稿集合。

进入 `main` 的内容需要同时满足：

1. 当前 M1 PRD 范围内。
2. 已经被负责人确认或校核。
3. 可以被项目直接运行、引用或查阅。
4. 不和权威线索表、ID 字典、校核报告冲突。

不进入 `main` 的内容包括：

1. 旧稿、早期推演、被明确排除的口径。
2. 未接入主线的小游戏原型。
3. M3 或其他墓葬的候选素材。
4. IDE 配置、日志、截图、压缩包和临时文件。
5. 没有被当前场景数据引用的大量备用素材。

## 四、分阶段执行

### 第一阶段：补仓库骨架

先完成低风险事项：

1. 新增 `README.md`。
2. 新增 `.gitignore`。
3. 新增整理方案和协作流程说明。
4. 不移动大量文件。

### 第二阶段：确定权威文档

由总负责人确认：

1. 哪份 PRD 是当前执行版本。
2. 哪份线索总表是权威版本。
3. 哪些旧稿必须归档，不能再被团队引用。
4. 哪些文档仍可作为参考。

### 第三阶段：文件归位

确认口径后再移动文件：

1. 将权威文档放入 `docs/PRD/`、`docs/handoff/`、`docs/team/`。
2. 将旧稿放入 `docs/archive/`。
3. 将旧数据放入归档目录。
4. 保持运行文件位置稳定，避免影响网页加载。

### 第四阶段：建立团队工作流

1. `main` 只由总负责人或整合人合并。
2. 日常整合建议使用 `develop`。
3. 模块开发使用 `feature/*`。
4. 小型整理使用 `task/*`。
5. 每次提交必须说明对应 PRD、线索 ID 或素材 ID。
6. 合并前检查运行、路径、文档口径和是否混入旧稿。

详细分支与文档分类见：

```text
docs/team/Git分支与文档分类.md
```

当前文档已按 `docs/PRD/`、`docs/team/`、`docs/handoff/`、`docs/assets/` 和 `docs/archive/` 初步归位。运行文件、`assets/` 和 `M1/` 原始资料母库暂时保持原位，避免影响网页运行和素材追溯。
