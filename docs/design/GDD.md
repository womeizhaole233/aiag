本 GDD 负责回答：

1. 玩家在游戏中具体做什么。
2. 每个章节如何推进。
3. 场景、线索、记录夹、NPC、小游戏如何互相连接。
4. 每个团队成员应交付什么内容。
5. 哪些内容可以进入当前 M1 主线，哪些必须暂缓或归档。

本文不是最终剧情稿，也不是完整技术方案。所有涉及真实考古事实、`clueId`、线索层级和结论边界的内容，以线索交付包为准。

## 3. 游戏概述

本项目是一款以白沙宋墓第一号墓 M1 为核心材料的沉浸式点击解谜游戏。玩家扮演考古实习生，通过观察墓葬图像、点击热区、收集线索、复查记录并形成章节结论，逐步理解 M1 的墓葬空间、图像内容、出土遗存和考古判断。

产品定位：

| 项目 | 内容 |
|---|---|
| 类型 | 考古题材沉浸式点击解谜游戏 |
| 玩家身份 | 考古实习生 |
| 核心体验 | 以墓葬图像为主视觉，玩家通过点击区域获得观察信息 |
| 玩法方向 | 点击探索 + 线索记录 + 复查推理 + NPC 剧情 + 轻量小游戏 |
| 风格 | 学术、科学、克制、轻悬疑、沉浸 |

必须避免：

1. 变成静态资料库。
2. 变成脱离材料的纯虚构机关解谜。
3. 用单点异常串成阴谋、灵异、案件或所谓核心真相。

## 4. 当前版本目标

当前版本只围绕 M1 主线，不扩展 M3，也不做完整多结局。

当前版本目标：

1. 跑通从墓外/环境到终章汇总的 M1 单线流程，墓外/环境作为当前最先进入 MVP 的入口章节。
2. 建立场景探索、线索收录、记录夹复查、章节结论的闭环。
3. 让固定 NPC 能在章节入口、关键线索、复查和收束处提供提示。
4. 第一版自由 NPC 接入 API，但必须按章节维护可引用线索和暂不回答话题。
5. 第一版小游戏严格控制为三个核心方向，三个小游戏优先级并列。
6. 所有模块交付都能被主项目或 AI 稳定接入。

暂不追求：

1. 完整 M3 开发。
2. 完整剧情分支。
3. 所有小游戏接入主项目。
4. 所有图片穷尽式使用。
5. 复杂多结局。
6. 自由 NPC 直接替代固定剧情、小游戏或记录夹完成关键推进。

## 5. 玩家体验目标

玩家体验应满足：

1. 玩家进入的是可探索的墓葬空间，而不是资料页面。
2. 玩家通过观察、点击、记录、复查逐步获得判断，而不是被直接告知答案。
3. 每个章节都有明确空间目标和推理目标。
4. 玩家能知道自己所在位置、朝向和前后空间关系。
5. NPC 提供现场氛围、工作提示和阶段性讨论，但不直接替玩家生成本章 R 结论。
6. 小游戏服务于考古动作，例如清理、辨读、定位。
7. 最终体验应让玩家感觉“我通过证据理解了这座墓”。

## 6. 核心玩法循环

主循环：

```text
进入场景
→ 观察全屏底图
→ 点击隐藏线索热区
→ 获得事实性观察文案
→ 线索自动进入记录夹
→ 在记录夹中复查、降级、汇总
→ 形成章节结论
→ 解锁下一空间或终章提示
```

NPC 与小游戏必须插入这个循环，不能另起一套无关流程。

示例：

```text
观察墓门
→ 收集门额、封门砖、门额背面彩画
→ NPC 提醒复查门的结构
→ 获得墓门阶段判断
→ 进入甬道
```

章节内循环：

```text
玩家搜集本章关键线索
→ 如果本章有小游戏，则进入对应小游戏
→ 小游戏产出清理 / 辨读 / 定位记录
→ 触发固定 NPC 剧情反馈
→ 玩家在记录夹整理证据
→ 生成本章 R 结论卡
→ 解锁下一章节
```

没有小游戏的章节仍然遵循“搜集线索 → 复查整理 → 触发剧情 → 生成 R 卡 → 解锁下一章”的流程。

### 7.2 终章线索墙与三栏报告

终章不新增新事实，只重组玩家已经获得的章节 R 卡、H 降级记录和 F 剧情记录。

线索墙区域：

| 区域 | 放置内容 | 通过条件 |
|---|---|---|
| 空间序列 | `R01`、`END-P0-01` | 六段空间顺序正确，并能标出后室局部 |
| 结构证据 | `R02`、`R03` | 墓门、甬道和顶部结构完成复查 |
| 图像功能 | `R04`、`R07` | 前室礼仪和后室寝居图像完成分类 |
| 文书遗存 | `R05`、`R06` | 题记、地券、人骨、铁钉完成叠合 |
| 已降级误读 | `R08`、`END-H-01` | H 线索至少完成降级或标为不能用于结论 |
| 剧情推测 | 剧情 F 线 | 只能放入“可推测 / 仍存疑”，不能放入“可判断” |

三栏报告：

| 栏目 | 可放入内容 | 需分栏或暂不直接放入 |
|---|---|---|
| 可判断 | 空间序列、墓门结构、甬道引导、前室礼仪展示、过道时间锚点、后室安葬核心与地下寝居图像 | 赵怀诚完整生平、死因、盗墓、谋杀、密室、机关 |
| 可推测 | F 剧情线中的姓名拼合、自述意图、时间差带来的阶段性安葬疑问 | 未经互证的官方史实、唯一真相、动机定论 |
| 仍存疑 | “永安”性质、隐字完整释读、迁葬原因、初葬对象、绢帛残损处补读 | 被 H 线索降级后仍强行串联出的阴谋 |

## 8. 场景探索系统

### 8.1 场景表现

1. 场景底图直接作为主视觉，尽量全屏展示。
2. 不使用资料库式卡片界面承载主体验。
3. 信息面板默认隐藏，只在玩家主动查看时展开。
4. 场景可以包含多个 `view`，例如前室东西南北壁、后室多墙面和顶部。
5. 近景视图必须支持返回上一视角。
6. 已进入区域回退后不应要求玩家重复探索。

### 8.2 热区类型

| 热区类型 | 显示方式 | 是否写入记录夹 | 用途 |
|---|---|---|---|
| 普通线索热区 | 默认隐藏 | 是 | 收录真实观察线索 |
| 近景热区 | 默认隐藏或弱提示 | 视内容而定 | 打开局部证据图或细节图 |
| 导航热区 | 悬停显示微透明文字 | 否 | 切换视角或进入下一空间 |
| 返回热区 | 悬停显示微透明文字 | 否 | 返回上一视角 |
| 查看位置 | 作为导航型热区或按钮 | 否 | 打开定位图 |

### 8.3 定位导航

定位导航图默认隐藏，玩家主动点击后弹出。定位图应显示：

1. 当前所在区域。
2. 当前面向的墙面或观察对象。
3. 已进入区域。
4. 已完成区域。
5. 墓门、甬道、前室、过道、后室之间的位置关系。

待确认：

1. 定位图最终 UI 形式。
2. 是否在所有章节都开放“查看位置”。
3. 墓外阶段与墓内空间图如何衔接。

## 9. 线索系统

线索来源必须以 M1 图片、线索表和映射表为准。游戏文案可以简短改写，但不能改变事实含义。

当前权威线索入口：

```text
docs/handoff/线索交付文档/01_核心交付/M1线索总表_PRD适配版_v1.0.md
docs/handoff/线索交付文档/06_ID与校核交付/M1线索ID字典与变更记录_v1.0.md
```

### 9.1 线索层级

| 层级 | 含义 | 使用规则 |
|---|---|---|
| P0 | 核心推进线索 | 可作为章节推进和核心结论依据 |
| P1 | 关键解释线索 | 支撑章节结论，通常不单独卡关 |
| P2 | 沉浸观察线索 | 增加现场感，不影响主线推进 |
| H | 误导或待验证线索 | 必须复查、降级或排除 |
| E | 环境线索 | 用于空间定位和章节进入 |
| R | 推理关系或章节结论 | 章节完成后生成结论卡或关系链 |
| F | 剧情虚构线索 | 只进入剧情线、可推测栏或存疑栏，不覆盖考古结论 |

### 9.2 线索进入主线的条件

一条线索成为必需线索，需要同时满足：

1. 来自真实 M1 材料，能够追溯到图片、线索表或映射表。
2. 能支撑章节核心判断或下一场景解锁。
3. 玩家缺少它时，会明显影响对当前章节的理解。

判断标准：

```text
如果拿掉这条线索，章节结论不成立，它应为 P0 或 P1。
如果拿掉这条线索，章节结论仍成立，它应为 P2、H 或 E。
如果它只服务剧情情绪，不服务考古判断，它应为 F。
```

### 9.3 剧情与证据分栏

为了给剧情负责人保留创作空间，当前不再把所有高风险推测写成“禁止写法”。第一版只设置三条硬性禁止：

1. 把题记一条线索写成解释整座墓的答案。
2. 把妇人启门、假门写成真实通道或密室入口。
3. 让 NPC 直接告诉玩家本章 R 结论。

其他涉及 F 线索、H 线索、人骨、铁钉、地券、身份、死因、迁葬、盗扰、官方史实等内容，可以作为人物猜测、剧情悬念、可推测或仍存疑处理。

处理规则：

1. 如果是已确认事实，必须能追溯到 PRD、线索总表或图片材料。
2. 如果是剧情发挥，必须标为 F、可推测、仍存疑或待确认。
3. 如果是 H 线索引出的误读，可以让角色讨论，但进入结论前必须复查、降级或保留存疑。
4. 如果后续补充了新的材料或主策划确认，可再升级为正式线索或章节结论。

## 10. 记录夹与复查系统

记录夹是把“点击图片”变成“解谜推理”的关键系统。

### 10.1 记录夹功能

记录夹需要支持：

1. 自动收录玩家点击获得的观察记录。
2. 以卡片列表查看线索，卡片显示 `clueId`、章节、层级、状态、绑定图片和下一步动作。
3. 以关系图查看线索之间的 R 链关系。
4. 在终章生成“可判断 / 可推测 / 仍存疑”三栏报告。
5. 复查多条相关线索。
6. 将 H 线索降级、排除或转为存疑。
7. 汇总章节结论。
8. 提示玩家还缺哪些记录或分析动作。

### 10.2 记录夹 UI

第一版记录夹采用三种视图：

| 视图 | 用途 | 使用阶段 |
|---|---|---|
| 卡片列表 | 查看已收录线索、待复查线索、H/F/R 状态 | 全流程 |
| 关系图 | 连接 P0/P1/E/H/F 与章节 R 链 | 章节收束、终章 |
| 三栏报告 | 把结论放入可判断、可推测、仍存疑 | 终章 |

卡片状态建议：

| 状态 | 含义 | 可否进入 R |
|---|---|---|
| 已观察 | 玩家点击并记录了可见事实 | 通常不能直接进入 R |
| 待复查 | 需要拓片、侧光、辨读、归类、叠合或对照 | 不能 |
| 已复查 | 完成指定考古动作 | 可以作为候选 |
| 待验证 | H 或 F 需要谨慎处理 | 不能进入学术 R |
| 已降级 | 不能承担强解释 | 只能进入 H 降级链 |
| 可入结论 | P0/P1/E 经复查后可支撑 R | 可以 |
| 存疑 | 终章仍不能判断的问题 | 只进三栏报告的“仍存疑” |

### 10.3 复查规则

1. 复杂章节不应只依靠“点够线索”完成。
2. 前室、后室、终章应要求玩家完成复查、降级和汇总动作。
3. 缺项提示应写成玩家能执行的动作，而不是开发术语。
4. 导航行为不进入记录夹。
5. 自由 NPC 默认不写入记录夹，除非主策划确认。

### 10.4 H 线索降级动作

H 线索不是删除，也不是禁止剧情使用。它可以制造好奇、误读和讨论，但进入结论前必须让玩家完成“我为什么还不能这样定案”的复查动作。降级或存疑后仍保留在记录夹中，用于终章方法总结和 `R08` 红鲱鱼降级链。

| H 线索 | 所属章节 | 降级动作 | 降级后归属 |
|---|---|---|---|
| `ENV-H-01` | 墓外 | 拖入“口述 / 证据 / 待验证”分类，并标注不能坐实诅咒、灵异或案件 | 地方传闻 / 待验证 |
| `GATE-H-01` | 墓门 | 放大观察灰缝颗粒，并与周边灰缝和封门砖整体组织对照 | 封门砖保存或工艺细节 |
| `COR-H-01` | 甬道 | 与前后边线、通道方向和顶部整体纹样复查 | 顶部磨损或偏移细节 |
| `FRONT-H-01` | 前室 | 将尖鞋与服饰、人物组和礼仪图像整体归类 | 服饰细节 |
| `FRONT-H-02` | 前室 | 把单件器物放回宴饮/陈设组合再提交 | 器物组合细节 |
| `PASS-H-01` | 过道 | 对照悬挂位置、重力方向和整壁构图 | 装饰构图细节 |
| `PASS-H-02` | 过道 | 题记必须与窗格、顶部、东壁共同提交，不能一证独大 | 题记解释边界 |
| `HS-H-01` | 后室 | 检查刻款、层位和与地券/人骨/铁钉的关系 | 随葬器物分布 |
| `HS-H-02` | 后室 | 清点铁钉，框出葬具范围，并与人骨和砖床叠合 | 葬具范围复查 |
| `HS-H-03` | 后室 | 对照北壁整体、门框、墙面和假门测绘图 | 图像母题 / 结构边界 |

### 10.5 R 结论卡生成

R 结论卡由玩家整理线索后生成，不是玩家手写论文。

流程：

```text
玩家把线索卡放入指定证据组
→ 游戏提出引导问题
→ 玩家提交一句或几句自己的判断
→ 系统判断是否与目标结论相似
→ 相似则自动生成正式 R 结论卡
→ 不相似则提示缺少哪类证据或哪项复查动作
```

实现口径：

1. 玩家提交内容只要与目标结论相似，即可获得对应 R 结论卡。
2. 游戏最终生成标准化 R 文案，保证记录夹和终章报告统一。
3. 判断方式可以先用关键词/选项匹配，后续再接语义相似度。
4. 失败反馈只提示缺少的证据组，不直接公布答案。

### 10.6 章节完成

章节完成应由以下内容共同判断：

1. 必需 P0/E 线索是否收集。
2. 必要 P1 是否完成复查。
3. H 线索是否被降级或标记为存疑。
4. 是否生成对应 R 结论。
5. 章节是否满足进入下一空间的条件。

待确认：

1. 每章 R 卡目标文案的相似度判断规则。
2. 记录夹关系图的具体视觉样式。
3. 三栏报告是否需要玩家手写摘要，还是只做拖拽分类。

## 12. 自由 NPC 对话系统

自由 NPC 指玩家主动点击 NPC 后，通过受控问答获得提示和陪伴。

核心原则：

```text
固定剧情负责推进，自由对话负责提示和陪伴。
```

### 12.1 第一版范围

第一版只建议开放一个 NPC：

| npcId | NPC | 位置 | 功能 |
|---|---|---|---|
| npc_professor | 粟柏年 | 记录夹或章节入口总览 | 回答当前目标、缺少什么、如何复查线索 |

其他 NPC 第一版可先只做固定剧情节点，后续再扩展自由对话。

### 12.2 自由 NPC 规则

1. 第一版实际接入 API。
2. 默认不写入记录夹。
3. 不能替玩家完成推理。
4. 不能凭空生成考古事实。
5. 不能提前剧透后续章节。
6. 必须引用线索表中允许引用的 `clueId`。
7. API Key 不得暴露在前端。
8. 不直接解锁场景，推进仍由线索、小游戏、固定剧情和 R 卡控制。

### 12.3 章节化自由 NPC 配置

自由 NPC 必须按章节单独维护 `allowedClues` 和 `forbiddenTopics`，避免把后续章节、F 线或 H 误读提前说成事实。

| 字段 | 说明 |
|---|---|
| chapterId | 墓外 / 墓门 / 甬道 / 前室 / 过道 / 后室 / 终章 |
| npcId | 第一版为 `npc_professor` |
| currentGoal | 本章玩家当前目标 |
| allowedClues | 本章允许引用的 `clueId` |
| forbiddenTopics | 本章不能回答或只能标存疑的话题 |
| completedR | 玩家已获得的 R 卡 |
| hintLevel | 轻提示 / 中提示 / 明确提示 |
| fallbackReply | API 失败或越界提问时的固定回复 |

### 12.4 自由 NPC 交付物

自由 NPC 负责人必须交付：

1. NPC 自由对话设定表。
2. NPC 知识范围表。
3. NPC 暂不透露 / 边界表。
4. NPC 默认回复规则。
5. 示例问答 10 条。
6. 按章节维护的 `allowedClues` 表。
7. 按章节维护的 `forbiddenTopics` 表。

技术/整合负责人需定义：

1. 自由对话入口位置。
2. API 调用上下文格式。
3. API 返回格式。
4. 前端显示方式。
5. 是否写入记录夹的规则。
6. 网络失败时的固定回复。

待确认：

1. 自由对话入口放在记录夹、章节入口还是工具按钮。
2. API 返回格式和安全代理由谁实现。
3. 每章 `allowedClues` 和 `forbiddenTopics` 的最终清单。

## 13. 小游戏接入系统

小游戏必须服务考古动作，不做无关机关。

### 13.1 第一版小游戏

三个小游戏优先级并列，都按“玩家搜集线索 → 进行游戏 → 触发剧情 → 记录夹整理 → 生成 R 卡”的章节循环接入。

| miniGameId | 小游戏 | 所属阶段 | 玩法方向 | 产出 |
|---|---|---|---|---|
| mg_dig_match3 | 挖土消消乐 | 墓外 / 墓门前 | 消除土块、碎砖、杂物，露出目标区域 | 清理记录，开放墓门区域或封门观察 |
| mg_inscription_reading | 题记辨读 | 过道 | 根据题记近景完成文字辨读或选择 | 题记辨读记录，确认 1099 年和“赵大翁”称谓 |
| mg_rear_relic_position | 后室遗物定位 | 后室 | 在分布图或后室平面中定位地券、人骨、铁钉等遗存 | 后室遗存关系记录，支撑后室汇总 |

暂不进入第一版开发：

1. 墓门文字解谜。
2. 甬道纹样校准。
3. 前室器物归类。
4. 文物擦灰。
5. 终章线索墙。
6. 反切拼字。

这些方向只可作为后续储备。

### 13.2 小游戏接入原则

1. 小游戏必须对应考古动作。
2. 小游戏完成后必须有明确产出。
3. 小游戏不能只为了好玩而存在。
4. 小游戏不能脱离 M1 主线独立存在。
5. 小游戏必须引用 `clueId` 或 `miniGameId`。
6. 小游戏失败后应允许重试或由 NPC 提示。
7. 小游戏难度应轻量，不应长期阻断玩家主线。

### 13.3 小游戏卡住机制

“卡住”指必须完成对应考古动作才能获得本章关键记录或 R 卡，不是让玩家长时间失败。

| miniGameId | 触发前必须已有 | 卡住位置 | 完成后触发 | 解锁 / 产出 |
|---|---|---|---|---|
| `mg_dig_match3` | `ENV-E-02`、`GATE-E-01` 或墓门前入口信息 | 进入墓门细节观察或封门砖记录前 | 陈怀远提示清理顺序和封门边界 | 开放 `GATE-P0-03` 或封门观察记录 |
| `mg_inscription_reading` | `PASS-P0-01`、`PASS-P1-01` | 生成 `PASS-R-01` 和进入后室前 | 周淼/粟柏年反馈题记是时间锚点，不是全部答案 | 题记辨读记录，确认 1099 年和“赵大翁”称谓 |
| `mg_rear_relic_position` | `HS-P0-01`、`HS-P0-02`、`HS-P0-03`、`HS-P0-04`、`HS-P1-05` | 生成 `HS-R-01` 和进入终章前 | 苏池/粟柏年反馈遗存叠合边界 | 后室遗存关系记录，支撑后室汇总 |

失败处理：

1. 允许重试。
2. 连续失败后由对应 NPC 给提示。
3. 提示只说明观察方法，不直接给结论。
4. 完成小游戏后必须写入记录夹，不能只播放剧情。

### 13.4 小游戏交付物

小游戏负责人必须交付：

1. 可独立运行小游戏原型。
2. 小游戏接入表。
3. 触发条件。
4. 成功条件。
5. 失败处理。
6. 完成后产出的线索或推进结果。
7. 引用的 `clueId`。
8. 使用素材。
9. 独立运行方式。

小游戏接入表字段：

| 字段 | 说明 |
|---|---|
| miniGameId | 小游戏编号 |
| 小游戏名称 | 挖土消消乐 / 题记辨读 / 后室遗物定位 |
| 所属章节 | 墓外 / 过道 / 后室 |
| 触发条件 | 玩家获得哪些线索后触发 |
| 入口方式 | 点击哪个热区、按钮或 NPC 节点 |
| 玩家操作 | 玩家具体做什么 |

4. 由什么触发。
5. 玩家做什么。
6. 完成后产出什么。
7. 是否影响主线推进。
8. 是否需要学术校核。

禁止只交：

1. 没有资产表的一堆图片。
2. 没有剧情节点表的对白。
3. 没有接入说明的小游戏 HTML。
4. 没有 `clueId` 的线索或 NPC 提示。
5. 没有结论边界的剧情推断。

## 16. 当前版本不做的内容

当前不进入 M1 第一版功能范围：

1. M3 场景制作和 M3 正式主线。
2. 独立小游戏合集。
3. 完整多结局。
4. 复杂剧情分支。
5. 自由 NPC 多角色聊天。
6. 自由 NPC 直接解锁关键场景。

以下内容可以出现在剧情讨论、人物猜测、可推测栏或仍存疑栏，但不直接进入“可判断”学术结论：

1. F 线索引出的身份、动机、时间差和个人自述。
2. H 线索引出的异常感、误读和复查讨论。
3. 人骨、铁钉、地券相关的身份、死因、迁葬、盗扰等推测。
4. 没有材料依据但服务剧情张力的疑问。

## 18. v0.1 验收标准

本 GDD v0.1 完成后，团队应能做到：

1. 主策划知道当前版本做什么、不做什么。
