# 白沙宋墓 M1 沉浸式点击解谜游戏 PRD（初步版）

> 本文用于整理当前项目的产品目标、玩法结构、内容范围与后续制作方向。未确定的内容只作为规划方向，不作为当前版本必须实现项。

## 1. 项目概述

本项目是一款以白沙宋墓第一号墓（M1）为核心材料的沉浸式点击解谜游戏。玩家扮演考古实习生，通过观察墓葬图片、点击图像热区、收集线索、完成记录夹复查与章节汇总，逐步理解 M1 的墓葬空间、图像内容、出土遗存和考古判断。

项目目标不是制作静态资料库，而是把真实考古材料转化为可探索、可记录、可推理、可叙事的互动体验。

当前项目已经形成从墓门到后室的主线原型，并开始规划 NPC 剧情互动与关卡小游戏。

## 2. 产品定位

| 项目 | 内容 |
|---|---|
| 产品类型 | 考古题材沉浸式点击解谜游戏 |
| 主要素材 | `D:\考古游戏\游戏第一版\M1` 中的图片、线索表、映射表 |
| 玩家身份 | 考古实习生 |
| 核心体验 | 以墓葬图像为全屏底图，玩家通过点击区域获得观察信息 |
| 玩法方向 | 点击探索 + 线索记录 + 复查推理 + NPC 剧情 + 轻量小游戏 |
| 风格关键词 | 学术、科学、克制、悬疑、沉浸 |
| 参考方向 | 有类似《纸嫁衣》的点击探索感，但内容依据真实考古材料 |

本项目需要避免两种偏差：

1. 变成资料库：只堆图片和说明，没有玩家行动。
2. 变成纯虚构解谜：脱离考古材料，过度制造悬疑和机关。

## 3. 目标用户

| 用户类型 | 需求 |
|---|---|
| 普通玩家 | 希望通过探索、点击、推理进入一个有悬疑感的墓葬空间 |
| 文博/考古兴趣者 | 希望在游戏中接触真实墓葬图像、术语和证据关系 |
| 课程/展示使用者 | 希望把白沙宋墓材料转化为可体验、可讲解的互动内容 |
| 创作者本人 | 希望把大量图片、线索表和映射表整理成有秩序的游戏关卡 |

## 4. 核心体验目标

玩家体验目标如下：

1. 玩家进入的不是资料页面，而是一座可探索的墓葬空间。
2. 玩家通过点击、记录、复查逐步获得判断，而不是被直接告知结论。
3. 每个章节都有明确的空间目标和推理目标。
4. 玩家能随时知道自己在墓葬中的位置、朝向和前后关系。
5. NPC 提供现场氛围、工作提示和阶段性讨论，但不直接告诉玩家本章 R 结论。
6. 小游戏服务于考古动作，例如清理、辨读、归类、拼合、定位。
7. 最终体验应让玩家感觉“我通过证据理解了这座墓”。

## 5. 世界观与玩家身份

玩家身份为考古实习生。玩家不是盗墓者，也不是全知研究者，而是在考古队工作流程中逐步学习观察、记录、复查和判断。

叙事基调为“真实考古工作 + 轻悬疑”。悬疑感来自空间未知、图像细节、材料之间的关系，而不是夸张灵异或无依据推断。

## 6. NPC 与剧情互动

后续计划加入类似橙光文游的剧情向互动。NPC 互动不应频繁打断普通探索，主要出现在章节入口、关键复查、章节收束和终章汇总时。

| NPC | 出场阶段 | 主要作用 | 语言风格 |
|---|---|---|---|
| 附近村民 | 墓外、未进入墓室前 | 提供地方记忆、发现经过、传闻氛围 | 口语、简短、带不确定性 |
| 考古技工 | 墓门、甬道、结构观察阶段 | 提醒封门、砖缝、构件、施工痕迹 | 经验型、直接、重现场 |
| 考古队员 | 拍照、编号、记录、复查阶段 | 提醒补拍、对照、归档 | 同伴协作，不直接给答案 |
| 考古领队（教授） | 章节收束、证据讨论、终章 | 引导玩家把线索组合成判断 | 学术、克制、边界清楚 |

NPC 互动建议采用“短对话 + 选择项 + 任务反馈”的形式：

1. 章节进入前：说明当前工作目标。
2. 关键线索后：提示玩家回看或对照相关材料。
3. 章节完成前：提醒打开记录夹进行复查、降级或汇总。
4. 章节完成后：给出阶段性判断，承接下一空间。

第一版 NPC 不需要复杂分支。玩家选择可以影响对话反馈，但不改变真实考古结论。

## 8. 核心玩法循环

```text
进入场景
→ 观察全屏底图
→ 点击隐藏线索热区
→ 获得事实性观察文案
→ 线索自动进入记录夹
→ 玩家在记录夹中复查、降级、汇总
→ 形成章节结论
→ 解锁下一空间或终章提示
```

NPC 与小游戏应插入这个循环，而不是另起一套无关流程。

例如：

```text
观察墓门
→ 收集门额、封门砖、门额背面彩画
→ NPC 提醒复查门的结构
→ 获得墓门阶段判断
→ 进入甬道
```

## 9. 场景探索系统

### 9.1 场景表现

1. 场景底图应直接作为主视觉，尽量全屏展示。
2. 不使用资料库式卡片界面承载主体验。
3. 不设置常驻顶部栏、常驻右侧面板或多余工具按钮。
4. 信息收集栏默认隐藏，只在玩家主动查看时展开。

### 9.2 热区规则

| 热区类型 | 显示方式 | 是否写入记录夹 |
|---|---|---|
| 普通线索热区 | 默认隐藏 | 是 |
| 近景热区 | 默认隐藏或弱提示 | 视其是否为真实线索而定 |
| 导航热区 | 鼠标悬停时显示微透明文字 | 否 |
| 返回热区 | 鼠标悬停时显示微透明文字 | 否 |
| 查看位置 | 可作为导航型热区 | 否 |

### 9.3 多视角与近景

一个场景可以包含多个 view。例如：

- 前室：入口总览、西壁、东壁、南壁、北壁、顶部。
- 后室：入口总览、北壁、南壁、西南壁、东北壁、西北壁、顶部。
- 甬道：顶部、东壁、西壁。

近景图必须支持返回上一视角。已进入的区域回退后不应要求玩家重复探索。

### 9.4 定位导航

定位导航图默认隐藏，玩家点击“查看位置”后弹出。定位图需要显示：

- 当前所在区域。
- 当前面向的墙面或观察对象。
- 已进入区域。
- 已完成区域。
- 墓门、甬道、前室、过道、后室之间的位置关系。

当前定位图第一版使用 `M1/00_墓葬全景与结构图/一号墓平剖面图.png`。

## 10. 线索系统

线索来源应以 `D:\考古游戏\游戏第一版\M1` 中的图片、线索表和映射表为准。游戏文案可以直接使用材料原文，也可以在不改变事实含义的前提下进行简短改写。

线索表是本项目的核心交付物之一。PRD 中只规定线索分类、字段、筛选规则和验收标准；具体每条线索的内容应单独整理成《M1线索总表》，由专人负责维护。

线索表需要解决三个问题：

1. 哪些线索是完成当前单线流程所必需的。
2. 哪些线索只作为补充观察、沉浸细节或待验证线索。
3. 哪些内容属于剧情创作，不能混入真实考古证据链。

### 10.1 线索分级

| 层级 | 含义 | 示例 |
|---|---|---|
| P0 | 核心推进线索 | 门额背面彩画、纪年题记、妇人启门、地券、人骨与铁钉 |
| P1 | 关键解释线索 | 破子棂窗、顶部宝盖、前室铺作、后室高几 |
| P2 | 沉浸观察线索 | 墙面残损、色层、砖缝、局部器物 |
| H | 误导或待验证线索 | 尖鞋、手部断口、流苏方向、单一异常 |
| E | 环境线索 | 平剖面图、入口、墓道、顶部结构图 |
| R | 推理关系链 | 章节结论、跨章节关系、终章汇总 |
| F | 剧情虚构线索 | 银扣、暗格、铜筒、绢帛、屏风藏字等剧情线内容 |

### 10.2 单线流程线索筛选规则

当前版本暂时只做一条流程线，不区分复杂主线和支线。线索表只需要判断每条线索在当前流程中的作用。

一条线索成为“必需线索”，需要同时满足以下条件：

1. 来自真实 M1 材料，能够追溯到图片、线索表或映射表。
2. 能支撑章节核心判断或下一场景解锁。
3. 玩家缺少它时，会明显影响对当前章节的理解。

补充线索仍然可以使用，但不应卡住玩家推进。

| 类型 | 当前流程作用 | 使用方式 |
|---|---|---|
| P0 | 必需 | 章节推进和核心结论 |
| P1 | 重要 | 支撑章节结论，通常不单独卡关 |
| P2 | 补充 | 增强现场感和可探索性 |
| H | 待验证 | 在记录夹中复查、降级或排除 |
| E | 环境 | 用于空间定位和章节进入 |
| F | 剧情 | 只服务剧情和悬疑，不作为考古证据 |

判断标准：

```text
如果拿掉这条线索，章节结论不成立，它应为 P0 或 P1。
如果拿掉这条线索，章节结论仍成立，它应为 P2、H 或 E。
如果它只服务剧情情绪，不服务考古判断，它应为 F。
```

### 10.3 记录夹机制

记录夹用于保存玩家已获得的真实线索，并支持后续复查。

当前记录夹需要承担：

1. 自动收录观察记录。
2. 复查多条线索。
3. 将误导或单点异常降级。
4. 汇总章节结论。
5. 提示玩家还缺哪些记录或分析动作。

前室和后室等复杂章节不应只依靠“点够线索”完成，还应要求玩家在记录夹中完成复查、降级与汇总。

## 11. 记录夹与分析系统

记录夹是把“点击图片”变成“解谜游戏”的关键系统。

当前项目已经形成以下方向：

| 操作 | 作用 |
|---|---|
| 收录 | 玩家点击真实线索后自动保存 |
| 复查 | 把多条相关线索重新放在一起看 |
| 降级 | 把单点异常处理为服饰细节、残损、局部现象等 |
| 汇总 | 形成章节判断或结论卡 |

体验要求：

1. 玩家尝试进入下一章但条件不足时，应提示缺少哪些观察或记录夹动作。
2. 缺项提示应写成玩家能找到的行动短句，例如“记录夹：完成后室汇总”。
3. 不使用“主线、二级细节、材料表”等开发用语作为玩家提示。
4. 导航行为不进入记录夹。

## 12. 小游戏系统

项目计划加入关卡小游戏，但小游戏必须与考古动作绑定，不做无关机关。

第一版小游戏数量需要严格控制，暂时只保留三个核心小游戏。其余小游戏只写方案，不进入当前开发。

### 12.1 设计原则

1. 小游戏服务于考古动作，例如清理、辨读、拼合、归类、定位。
2. 小游戏应产出线索、近景图、章节凭证或 NPC 反馈。
3. 小游戏难度应轻量，不应长期阻断玩家主线。
4. 失败后允许重试，或由 NPC 给出提示。
5. 小游戏产出的重要信息应进入记录夹。

### 12.2 已确定方向

| 小游戏 | 适用阶段 | 玩法方向 | 产出 |
|---|---|---|---|
| 挖土消消乐 | 前段：墓外 / 墓道初清 / 墓门前 | 消除土块、碎砖、杂物，逐步露出目标区域 | 墓道初清记录，开放墓门区域 |
| 题记辨读 | 中段：过道 | 根据题记近景完成文字辨读或选择 | 纪年题记辨读记录，支撑进入后室 |
| 后室遗物定位 | 后段：后室 | 在分布图或后室平面中定位地券、人骨、铁钉等遗存 | 后室遗存关系记录，支撑后室汇总 |

第一版三个小游戏的优先级为：

1. 挖土消消乐：负责前段“进入现场”的操作感。
2. 题记辨读：负责中段“确认文字证据”的操作感。
3. 后室遗物定位：负责后段“汇总遗存关系”的操作感。

墓门文字解谜、甬道纹样校准、前室器物归类、终章线索墙暂时只作为后续方案。

### 12.3 可继续构思方向

| 场景 | 可考虑小游戏 | 绑定线索 |
|---|---|---|
| 甬道 | 东西壁图像朝向对照、题字位置比对 | L10 两壁对照、题字高度 |
| 前室 | 礼仪器物归类 | 女乐、注子、高瓶、壁函、铺作 |
| 过道 | 纪年题记辨读、窗格结构复原 | 纪年题记、破子棂窗、顶部宝盖 |
| 后室 | 日常器物归类 | 镜台、盆架、剪刀熨斗 |
| 终章 | 线索墙连线 | 各章节结论卡与关系链 |

这些方向不进入第一版开发，只进入后续设计储备。

## 14. 图片资产规划

图片素材来源为 `D:\考古游戏\游戏第一版\M1`。项目中使用的图片应逐步复制或整理到 `assets/M1`，并在数据文件中统一引用。

图片应按用途分类：

| 类型 | 含义 |
|---|---|
| 主底图 | 一个场景或视角的基础图 |
| 近景图 | 点击某个区域后进入的局部观察图 |
| 证据图 | 用于记录夹、结论卡或复查的关键证据 |
| 备用图 | 内容重复、角度不稳定或暂时没有使用位置的图片 |
| 补充图 | 当前缺少但体验上需要的总览图或过渡图 |

当前仍需长期维护：

1. 已进入项目的图片清单。
2. 原始资料夹中尚未使用的图片清单。
3. 每张图片是否已被 `data/scenes.js` 引用。
4. 缺失总览图的补图需求，例如甬道总交互图、墓外环境图。

## 15. UI 与交互规范

当前 UI 方向已经明确：

1. 图片底图是主界面，不增加多余装饰。
2. 不做常驻顶部栏。
3. 不做常驻右侧信息面板。
4. 信息收集栏默认隐藏。
5. 普通线索热区隐藏。
6. 导航、返回、转向热区仅悬停显示微透明文字。
7. 弹窗只承载观察事实。
8. 定位图只在玩家主动查看时出现。
9. 玩家进入下一场景后，仍应能返回已进入空间。
10. 已完成区域回退后不要求重复探索。

## 16. 线索文案规范

线索文案必须遵守《线索文案规范.md》。

基本要求：

1. 只描述图像中可观察到的事实。
2. 优先写位置、形态、尺寸、材质、色层、保存状态、空间关系。
3. 弹窗控制在 2 至 3 句。
4. 记录文案比弹窗更短。
5. 不使用“材料表显示”“当前作为主线线索”“二级细节”等开发说明。
6. 不直接替玩家生成本章 R 结论。
7. 不写过度推断句。
8. 不使用翻译腔或过度文学化表达。

推理性内容应放在记录夹复查、NPC 讨论、结论卡或终章汇总中。

## 17. 结论与推理系统

每个章节都应有一个核心推理目标。玩家不是单纯收集事实，而是通过事实形成阶段判断。

| 章节 | 结论方向 |
|---|---|
| 墓门 | 墓门既是入口，也是第一道证据界面 |
| 甬道 | 甬道承担过渡与引导，东西壁与顶部图像共同建立通道秩序 |
| 前室 | 前室是礼仪展示空间，并存在年代错层或后期处理问题 |
| 过道 | 过道集中出现时间信息和空间转换信息 |
| 后室 | 后室集中墓主遗存、文书、图像核心与日常器物线 |
| 终章 | M1 的意义来自空间、图像、文书、遗物等多层材料共同指向 |

结论卡不应出现在普通弹窗中，而应在玩家完成章节条件后生成或展示。

## 18. 章节完成条件

章节完成条件需要同时考虑三类内容：

1. 是否获得必要观察记录。
2. 是否完成记录夹中的复查、降级或汇总。
3. 是否完成必要小游戏或剧情节点。

当前已经明确的完成方向：

| 章节 | 当前完成逻辑 |
|---|---|
| 墓门 | 收集指定墓门线索后开放甬道 |
| 甬道 | 完成核心甬道观察后开放前室，L10 作为支线观察 |
| 前室 | 完成关键墙面观察，并在记录夹中完成前室复查、降级、汇总 |
| 过道 | 纪年题记、破子棂窗、顶部宝盖三项作为进入后室条件 |
| 后室 | 核心证据 + 日常器物线共同构成后室完成条件 |
| 终章 | 后室完成后出现终章汇总占位提示 |

## 19. 技术与数据要求

该部分只记录当前产品需求层面的技术约束，不展开实现细节。

1. 场景数据与游戏逻辑分离。
2. 图片路径优先统一到 `assets/M1`。
3. 每条证据热区应保留 `sourceFile` 与 `sourceClueId`。
4. 同一场景应支持多个 view。
5. view 应支持进入 closeup 近景图并返回。
6. 导航热区不进入记录夹。
7. 已完成场景回退后保持完成状态。
8. 需要支持定位图显示当前区域和朝向。

## 20. 团队模块交付格式

当前半个月周期内，各成员不应只交散乱素材或单独代码，而应按模块交付可被主项目和 AI 理解的表格与文件。

### 20.1 场景模块交付

场景模块负责把图片、线索、NPC、小游戏放进具体游戏空间。它不是图片编号本身，而是说明图片如何变成可玩的场景。

场景模块至少提交：

| 字段 | 说明 |
|---|---|
| sceneId | 场景编号，例如 `gate`、`corridor`、`front` |
| viewId | 视角编号，例如 `front_west`、`rear_overview` |
| imageId | 当前视角使用哪张图片 |
| imagePath | 图片在项目中的路径 |
| hotspotId | 热区编号 |
| hotspotType | clue / navigation / closeup / npc / miniGame |
| hotspot位置 | x、y、width、height 或百分比坐标 |
| 点击结果 | 弹线索、切换视角、打开剧情、进入小游戏 |
| clueId | 如果是线索热区，对应线索编号 |
| nodeId | 如果是剧情热区，对应剧情节点 |
| miniGameId | 如果是小游戏热区，对应小游戏编号 |
| 返回关系 | 从近景或子视角返回哪里 |
| 解锁条件 | 需要哪些线索、剧情或小游戏完成 |
| 缺项提示 | 条件不足时提示玩家做什么 |

### 20.2 NPC 固定剧情模块交付

固定剧情 NPC 负责人不应只提交长篇剧情文本，需要提交可接入游戏的剧情节点表。固定剧情指游戏中确定会发生的章节进入、任务说明、玩家选择、小游戏前后引导和章节收束。

NPC 固定剧情模块至少提交三类内容：

1. NPC 角色表。
2. 固定剧情节点表。
3. NPC 与线索绑定表。

NPC 角色表字段：

| 字段 | 说明 |
|---|---|
| npcId | NPC 编号 |
| NPC 名称 | 例如粟柏年、周淼、赵老倔 |
| 身份 | 教授 / 队员 / 技工 / 村民 |
| 出场章节 | 墓外、墓门、甬道、前室等 |
| 功能 | 引导、提示、复查、收束、氛围 |
| 语言风格 | 学术、口语、经验型、同伴式 |
| 是否虚构角色 | 是 / 否 |

剧情节点表字段：

| 字段 | 说明 |
|---|---|
| nodeId | 剧情节点编号 |
| 所属章节 | 墓外 / 墓门 / 甬道 / 前室等 |
| 触发条件 | 首次进入、点击线索后、小游戏前、章节完成后 |
| 出场 NPC | npcId |
| 节点目的 | 说明目标 / 提醒观察 / 选择判断 / 收束结论 |
| 对话正文 | NPC 与玩家对话 |
| 玩家选项 | A/B/C 或无选项 |
| 选项反馈 | 每个选项对应的反馈 |
| 绑定线索 | clueId |
| 产出结果 | 新增记录 / 解锁小游戏 / 解锁场景 / 仅剧情 |
| 是否影响推进 | 是 / 否 |
| 是否虚构剧情 | 是 / 否 |

NPC 与线索绑定表用于防止 NPC 说出无依据内容：

| 字段 | 说明 |
|---|---|
| nodeId | 剧情节点编号 |
| npcId | NPC 编号 |
| 绑定线索 | clueId |
| NPC 可以说什么 | 基于线索可表达的内容 |
| NPC 不能说什么 | 不应提前剧透或无依据推断的内容 |

### 20.2.1 NPC 自由对话模块交付

自由对话 NPC 负责人负责设计“玩家主动点击 NPC 并输入问题”时，API 应如何扮演该 NPC。该模块不写长篇剧情，而是写知识范围、暂不回答边界和回复规则。

NPC 交互分为固定剧情和自由对话两类：

| 类型 | 作用 | 是否控制流程 |
|---|---|---|
| 固定剧情 NPC | 章节进入、关键节点、选项反馈、章节收束 | 可以控制流程 |
| 自由对话 NPC | 玩家主动提问、轻提示、术语解释、线索复查 | 默认不控制流程 |

第一版自由对话建议只开放“考古领队 / 教授”一个 NPC，用于回答当前目标、缺少什么、如何复查线索。

NPC 自由对话模块至少提交四类内容：

1. NPC 自由对话设定表。
2. NPC 知识范围表。
3. NPC 暂不回答 / 边界表。
4. NPC 默认回复规则。
5. 示例问答至少 10 条。

自由对话运行逻辑：

```text
玩家点击 NPC 或“询问领队”
→ 输入问题
→ 系统收集当前章节、视角、已收集线索、缺失线索和 NPC 知识范围
→ 后端调用 API
→ 返回 NPC 口吻回答
→ 前端显示
→ 默认不写入记录夹，不直接解锁场景
```

自由对话 NPC 必须遵守：

1. 不能凭空生成考古事实。
2. 不能提前剧透后续章节。
3. 不能替玩家直接完成推理。
4. 默认不写入记录夹。
5. 如需写入，只能标为“NPC 提示”，不能标为“考古证据”。
6. API Key 不得写在前端代码中。

详细规则见《NPC自由对话模块需求文档.md》。

### 20.3 小游戏模块交付

小游戏组不应只提交独立 HTML 或代码文件。每个小游戏都必须说明如何接入当前单线流程。

第一版小游戏只保留三个：

| miniGameId | 小游戏 | 位置 |
|---|---|---|
| mg_dig_match3 | 挖土消消乐 | 墓外 / 墓门前 |
| mg_inscription_reading | 题记辨读 | 过道 |
| mg_rear_relic_position | 后室遗物定位 | 后室 |

小游戏模块至少提交：

| 字段 | 说明 |
|---|---|
| miniGameId | 小游戏编号 |
| 小游戏名称 | 挖土消消乐 / 题记辨读 / 后室遗物定位 |
| 所属章节 | 墓外 / 过道 / 后室 |
| 触发条件 | 获得哪些线索或进入哪个剧情节点后触发 |
| 入口方式 | 点击热区、NPC 对话后出现、章节收束前出现 |
| 玩家操作 | 玩家具体做什么 |
| 成功条件 | 如何判定完成 |
| 失败处理 | 是否允许重试，是否由 NPC 提示 |
| 产出结果 | 新增记录、解锁场景、解锁近景或仅剧情反馈 |
| 绑定线索 | 使用或产出的 clueId |
| 使用素材 | 图片、音效、字体等 |
| 独立运行方式 | 如何本地打开测试 |
| 接入备注 | 主项目需要提供什么数据或回调 |

### 20.3.1 小游戏交付验收条件

每个小游戏必须满足以下交付条件：

| 小游戏 | 必须交付 | 完成结果 |
|---|---|---|
| 挖土消消乐 | 可独立运行原型、方块素材、接入表、胜利条件说明 | 完成后产出 `dig_clear_record` 或同等清理记录 |
| 题记辨读 | 题记图片或拓片素材、可选字/辨读逻辑、接入表 | 完成后产出过道题记辨读记录 |
| 后室遗物定位 | 后室分布图或平面图、可定位对象、正确位置规则、接入表 | 完成后产出后室遗存关系记录 |

三个小游戏都需要说明：

1. 触发条件。
2. 入口位置。
3. 玩家操作。
4. 成功条件。
5. 失败或重试处理。
6. 完成后是否写入记录夹。
7. 完成后是否解锁下一节点。
8. 使用哪些 `clueId`。

小游戏不合格的情况：

```text
只有独立 HTML，没有接入表。
玩法和考古动作无关。
完成后没有记录、提示或推进结果。
自行创造不在线索表中的主流程线索。
需要玩家长时间反复尝试才能推进。
```

### 20.4 图片编号与场景接入的关系

`imageId` 是图片编号，只回答“这张图片是谁”。场景接入表回答“这张图片在游戏里如何被使用”。

示例：

```text
imageId: M1_gate_main
图片路径: assets/M1/01_墓门/第一号墓墓门.png
图片类型: 主底图
```

对应场景接入：

```text
sceneId: gate
viewId: gate_main
imageId: M1_gate_main
hotspotId: gate_lintel_hotspot
hotspotType: clue
clueId: gate_L01
点击结果: 弹出门额观察文案，并写入记录夹
```

## 21. 最小可运行 MVP 需要的模块

除了场景、剧情和小游戏，要保证最小 MVP 能运行，还需要以下基础模块：

| 模块 | 作用 | MVP 要求 |
|---|---|---|
| 图片资产模块 | 管理图片编号、路径和用途 | 每张已用图片有 imageId 和项目路径 |
| 线索模块 | 管理 clueId、文案、来源和章节关系 | 每个可点击线索有 clueId 和记录文案 |
| 场景导航模块 | 管理场景、视角、返回和前进 | 玩家能从入口推进并自由返回 |
| 记录夹模块 | 保存已获得线索 | 点击真实线索后能自动记录 |
| 进度状态模块 | 保存已完成线索、剧情、小游戏 | 回退后不重复要求，完成状态不丢失 |
| 条件判断模块 | 判断是否可进入下一节点 | 条件不足时提示缺什么 |
| 弹窗模块 | 显示线索、剧情、完成提示 | 文案清楚，不遮挡主体验 |
| 定位导航模块 | 显示玩家所在空间和朝向 | 玩家能知道自己在墓葬中的位置 |
| 小游戏接入模块 | 打开小游戏并接收完成结果 | 至少一个小游戏能完成并返回主流程 |
| NPC 自由对话模块 | 玩家主动询问 NPC 并获得受控提示 | 第一版只开放一个 NPC，默认不推进流程 |
| 存档模块 | 保存玩家当前进度 | 最低要求可用浏览器本地存储 |

最小 MVP 的目标不是内容完整，而是跑通一条闭环：

```text
进入场景
→ 触发 NPC 或观察目标
→ 点击图片获得线索
→ 线索进入记录夹
→ 触发一个小游戏
→ 小游戏完成后写入结果
→ 解锁下一场景
→ 玩家可以返回查看
```

## 22. 版本规划

### 第一阶段：稳定主线体验

目标：保证玩家能从墓门推进到后室，并触发终章占位。

重点：

- 检查墓门、甬道、前室、过道、后室完整流程。
- 校准关键热区。
- 统一图片路径。
- 确认返回自由度。
- 确认导航不写入记录夹。

### 第二阶段：补强空间与素材

目标：减少空间迷失，让每个章节像独立关卡。

重点：

- 完善墓外或环境阶段。
- 补缺失总览图。
- 整理未使用图片。
- 继续补前室和后室可用近景。

### 第三阶段：补强线索与结论

目标：让玩家知道线索最终指向什么。

重点：

- 完善每章结论卡。
- 建立跨章节关系链。
- 制作终章线索墙。
- 继续清理文案，使其符合规范。

### 第四阶段：加入 NPC 剧情互动

目标：用考古队角色增强叙事和引导。

重点：

- 墓外村民开场。
- 墓门技工提示。
- 各章节考古队员协助记录。
- 教授在章节收束时出现。

### 第五阶段：加入轻量小游戏

目标：让关键考古动作更有操作感。

重点：

- 挖土消消乐原型。
- 过道题记辨读原型。
- 后室遗物定位原型。
- 墓门文字解谜、甬道纹样校准、前室归类作为后续增强方案。

## 23. MVP 范围

MVP 应包含：

1. 墓门、甬道、前室、过道、后室主线流程。
2. 每章至少一个核心推理目标。
3. 全屏底图点击探索。
4. 自动记录线索。
5. 记录夹复查、降级、汇总的基础机制。
6. 定位图。
7. 终章占位提示。
8. 线索文案符合《线索文案规范.md》。
9. 一份可供团队和 AI 接入使用的《M1线索总表》。

MVP 暂不强制包含：

1. 完整墓外剧情。
2. 完整 NPC 分支。
3. 所有小游戏。
4. 完整终章线索墙。
5. 所有图片的穷尽式使用。
6. 墓门文字解谜、甬道纹样校准、前室器物归类等后续小游戏。

## 24. 验收标准

| 模块 | 验收标准 |
|---|---|
| 主线流程 | 玩家能从墓门推进到后室，并触发终章占位 |
| 返回机制 | 玩家能返回已进入的上一级空间或入口总览 |
| 场景体验 | 场景以图片底图为核心，不呈现资料库式界面 |
| 线索记录 | 点击真实线索后自动进入记录夹 |
| 导航记录 | 导航、返回、转向不进入记录夹 |
| 缺项提示 | 玩家尝试推进时能知道还缺哪些观察或记录夹动作 |
| 文案 | 无开发说明、无材料编号外显、无过度推理 |
| 定位 | 玩家能查看当前位置和朝向 |
| 前室 | 能完成礼仪空间与年代错层相关汇总 |
| 后室 | 能完成核心证据与日常器物线的综合汇总 |
| NPC 规划 | 明确 NPC 身份、出场阶段和叙事功能 |
| 小游戏规划 | 明确小游戏与考古动作、线索产出、章节推进的关系 |
| 线索总表 | 明确 P0/P1/P2/H/E/R/F 分类、是否进入主线、来源依据和记录文案 |
